# Sprint R4.1: Unit-Term Slug Migration

## Goal

Migrate unit `terms` fields from old Dutch display strings to canonical `begrippen.json` slug IDs where the mapping is exact and unique.

## Context

R3.2 completed reviewed prerequisite corrections. R4.1 now improves reference consistency by making unit-to-term links point at stable term IDs instead of mutable display text.

This is a protected-reference mutation sprint. Machine references may change only through a validated migration script or CLI path, never by hand.

## Operationalized sprint procedure

1. Establish baseline counts: unit count, live unit count, unique current unit term strings, and existing term registry count.
2. Build a deterministic mapping from unit term strings to term IDs using exact normalized matches against term ID, canonical `term_nl`, synonyms, deprecated forms, and safe gloss-stripped forms.
3. Classify mappings as safe only when exactly one live term ID matches.
4. Write unresolved and ambiguous mappings to the report. Do not create new slugs in R4.1.
5. Update the unit validator so canonical term IDs are valid unit `terms` references.
6. Dry-run the migration and inspect safe/unresolved counts.
7. Apply only safe mappings through the migration script.
8. Regenerate `micro-teaching-units.json`, `terms-coverage.md`, `begrippen.json`, `begrippen.md`, and `begrippen-coverage.md`.
9. Write result, diff summary, and result JSON.
10. Update the roadmap: R4.1 complete, unresolved terms routed to later term-registry cleanup.
