## Operationalized sprint procedure

1. Read `review-packet.md`, all three subagent review JSON files, the current unit catalog, and the R2.3 baseline.
2. Check whether the audit/review packet counts still match the current catalog. If they do not, pause the interview, refresh the empty-needs audit and review packet, and validate the refreshed artifacts before asking gate questions.
3. Ask 8-10 calibration questions grounded in the current review packet, one question at a time. Do not dump the full interview list.
4. For each question, offer clear multiple-choice options where useful plus an open-answer option, and include unit names/descriptions when using unit IDs.
5. Record each human answer in `human-interview.md`.
6. Analyze answer patterns and choose one next mode:
   - `continue_targeted`
   - `batch_confirm_low_risk`
   - `pause_for_missing_evidence`
   - `rerun_audit_with_revised_rubric`
   - `schema_change_required`
   - `propose_gate_closure`
7. Ask targeted follow-ups or batch-confirm low-risk rules based on the selected mode, still one question or decision at a time.
8. Draft a closure proposal that names the gate status, accepted classes of recommendations, held/rejected classes, allowed next sprint, blocked next sprint, and conditions.
9. Ask for explicit human confirmation of the closure proposal.
10. Only after explicit confirmation, write `human-interview.md`, `gate-closure.json`, R2.3 result files, and run the gate validator plus sprint acceptance tests.
11. Stop if the human reviewer does not confirm a closure proposal. Do not apply dependency corrections in R2.3.
