# Sprint R2.4: Evidence And Unit-Design Cleanup

## Goal

Produce a bounded, review-ready evidence and unit-design packet after `GATE-R2-empty-needs` closed as `hold`, so later CLI mutation work can proceed only from exact evidence and explicit decisions.

## Context

R2.1 generated the non-mutating empty-needs audit. R2.2 prepared pedagogy, evidence, and data-integrity review artifacts. R2.3 closed `GATE-R2-empty-needs` as `hold`: the audit is useful diagnostic input, but it does not authorize dependency mutation.

R2.4 is deliberately narrow. It finishes the job required by the hold decision without expanding Phase 2 into open-ended discovery. Remaining broad cleanup belongs to later phases once the mutation path and evidence anchors are stronger.

## Operationalized sprint procedure

1. Establish the baseline from R2.3: gate status `hold`, 190 live units, 61 empty-needs units, no protected reference mutation allowed.
2. Extract only the R2.4-required decision surfaces:
   - candidate elasticity prerequisite edges raised by the gate
   - `D04 Elasticiteit en goederenclassificatie` unit-design scope
   - foundational A-domain zero-needs classification
   - labor-market and unemployment sequencing second pass
   - rejected market-graph machine suggestions `D10 -> D01/D03`, `D13 -> D03`, and `D32 -> D01/D03`
3. Attach exact evidence only from current exam-question records or target-exercise records. Use syllabus text only as context, not as proof for mutation.
4. Classify every included item as one of:
   - `review_ready`: evidence is attached, but human review and CLI mutation are still required
   - `unit_design_required`: no edge should be applied until unit scope is changed
   - `rejected`: do not apply this machine suggestion unless new evidence overturns it
   - `hold_for_later_phase`: useful issue, but outside R2.4's bounded scope
5. Write the packet as a decision queue, not as a mutation. Include evidence paths, exact anchors, next action, and proof required to close.
6. Build a deterministic validator that proves the packet exists, references real files, includes the required R2.4 surfaces, and declares no protected reference data changes.
7. Run sprint and packet validators. Stop if the packet attempts to authorize mutation, touches protected reference data, or expands into broad cleanup.
8. Complete sprint result artifacts and update the roadmap to mark R2.4 complete while keeping R3.2 blocked until human review plus R3.1 CLI/documentation readiness.
