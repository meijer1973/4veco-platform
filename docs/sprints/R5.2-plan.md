# Sprint R5.2: Alignment Graph

## Goal

Create the first alignment graph projection from the validated evidence-anchor layer, with explicit relation types and review status.

## Context

R5.1 created evidence anchors and claims. R5.2 may now project those claims into graph edges, but the graph must remain non-authoritative until the R5.3 review gate closes.

The graph is a data product, not a pedagogical decision maker. It must only contain edges that trace back to evidence IDs and claims.

## Operationalized sprint procedure

1. Establish baseline: evidence anchors exist and validate; no alignment graph data file exists yet.
2. Expand `alignment-edge.schema.json` with namespaced node IDs, bounded relations, evidence IDs, claim ID, strength, source layer, and review status.
3. Build graph edges only from claims in `references/data/evidence-anchors.json`.
4. Create `prerequisite` edges for `unit_prerequisite` claims, `derived_from` edges from claims to evidence anchors, and narrow support edges for term-definition/source-policy/diagnostic claims.
5. Mark every graph edge as `pending_r5_3_review`; no edge may be marked authoritative in R5.2.
6. Validate unique edge IDs, allowed relation types, source claim references, evidence references, no missing nodes, and no generated-report edge marked primary or authoritative.
7. Write integrity reports, result artifacts, and roadmap update. Stop before retrieval, diagnostics, or R5.3 gate closure.
