# Sprint R8.1: Scoped QC Issue Model

## Goal

Create a lightweight, internal-only QC issue model that improves reference and RAG trust without becoming a general issue tracker or curriculum authority.

## Context

The roadmap places R8.1 immediately after the RX.2 first-lane mutation work. The data foundation now has machine units, JSON-first reports, reference health, retrieval chunks, owned-content projection, exercise overlays, and several gate conditions that need durable issue tracking.

R8.1 must stay narrow. It should model only the categories needed now: reference quality, evidence sufficiency, unit design, extraction integrity, report drift, source-version drift, term-link gaps, needs gaps, and production-readiness warnings.

The issue records are internal governance data. They can support dashboards and RAG retrieval, but they are diagnostic signals, not primary evidence and not curriculum authority.

## Operationalized sprint procedure

1. Record this plan and baseline before changing the issue model. Stop if R8.1 is not the immediate next sprint in the roadmap.
2. Tighten `quality-issue.schema.json` around the R8.1 category set, severity, owner/team, affected surface, evidence references, next action, proof required to close, and explicit internal-only authority boundary.
3. Create an internal seed issue log in `references/data/qc/` using current repository evidence. Stop if an issue would require treating generated reports as primary proof.
4. Add a read-only validator that checks issue IDs, categories, statuses, evidence paths, owner/team, proof-to-close text, and the non-authoritative boundary.
5. Extend JSON-first reports with `reference-quality-issues` and include it in report validation.
6. Add a reference-health hook that exposes QC issue counts while preserving `internal_only: true`, `curriculum_authority: false`, `student_facing_exposure: false`, and `primary_evidence: false`.
7. Regenerate reports, reference health, RAG chunks, retrieval evals, source manifest, document inventory, gate URL index, and roadmap version index.
8. Run validators and acceptance tests. Stop if the issue model implies curriculum authority, student-facing use, protected-source mutation, or a general-purpose issue-tracker scope.

## Rollback plan

Revert the R8.1 commit to remove the schema tightening, QC issue log, validator, generated reports, dashboard hook, roadmap update, and regenerated artifacts.

No protected reference data rollback should be needed because R8.1 must not edit `references/machine/` or `references/external/`.

## Human review required

No full human gate is required for R8.1. This sprint is an internal hardening sprint with bounded categories and validators. Human review is required later if a follow-up proposes to promote QC issues into production gates, student-facing use, curriculum authority, or machine-registry mutation.
