# Sprint R4.4: Micro-Teaching Unit Quality Packet

## Goal

Create a non-mutating quality packet for the remaining micro-teaching-unit issues so the next mutation sprint can apply only reviewed, evidence-backed changes through CLI.

## Context

R3.2 applied the previously approved empty-needs corrections. The current catalog still has live units without `needs`, live units without term links, unresolved unit term strings, and blueprint missing-unit flags. The user wants unit quality moved forward before broad owned-source/RAG expansion.

## Allowed paths

- `docs/sprints/R4.4-plan.md`
- `references/data/sprints/R4.4.plan.json`
- `references/data/sprints/R4.4.result.json`
- `reports/sprints/R4.4-baseline.md`
- `reports/sprints/R4.4-result.md`
- `reports/sprints/R4.4-diff-summary.md`
- `reports/review-gates/GATE-R4-micro-unit-quality/R4.4-micro-unit-quality-packet.md`
- `reports/review-gates/GATE-R4-micro-unit-quality/R4.4-micro-unit-quality-packet.json`
- `build-scripts/references/build-micro-unit-quality-packet.js`
- roadmap status updates in `references/reference-team-roadmap.md`
- regenerated reference inventories if new report artifacts are added

## Forbidden paths

- hand edits to `references/machine/`
- hand edits to `references/external/`
- applying unit mutations in R4.4
- adding opportunistic dependency edges
- minting units from syllabus text alone
- student diagnostics
- adaptive routing
- student-facing AI
- automatic sequencing, mastery, or summative decisions

## Outputs

- A machine-readable R4.4 packet with explicit issue queues and recommended next actions.
- A human-readable R4.4 packet summarizing what can move to mutation review, what needs evidence/design review, and what remains deferred.
- Sprint result and diff summary.
- Roadmap updated to show R4.4 completed and R4.5 as the next mutation sprint.

## Operationalized sprint procedure

1. Establish baseline counts for empty-needs units, missing term links, unresolved term strings, and blueprint missing-unit flags from current generated reports.
2. Build the packet from existing reports and registries only. Stop if a required source report is missing or stale enough that the packet would misrepresent the catalog.
3. Classify each candidate as reviewed non-mutation, review-before-mutation, term-link review, term-registry review, unit-design review, existing-unit-match review, duplicate, or deferred.
4. Do not edit `references/machine/` or `references/external/`; R4.4 is a review packet sprint only.
5. Validate JSON shape, regenerate inventories, record acceptance tests, and update the roadmap so R4.5 is the next CLI-only mutation step.

## Rollback plan

If the packet is wrong or too broad, revert the R4.4 commit. Since R4.4 does not mutate machine references, rollback only removes the packet, sprint records, and generator script.

## Human review required

No formal gate closure is required for R4.4 itself. Human review is required before R4.5 may apply any recommended machine-reference mutations.
