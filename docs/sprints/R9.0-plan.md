# Sprint R9.0: Legacy Module-3 Reference Minimization

## Goal

Minimize non-functional references to Module 3 as a conceptual anchor in current platform/reference guidance, while preserving legacy functional paths and frozen-target guardrails.

## Context

The platform direction is now markdown-native book/source material in `../4veco-lessen` and the reference/RAG corpus. Module 3 remains operationally frozen until September 2026, but it should no longer be used as the default example or course-design backbone for new work. This cleanup happens before R9.1 so the owned-source registry starts from current course and blueprint terminology instead of legacy Module 3 terminology.

## Allowed paths

- `docs/sprints/R9.0-plan.md`
- `references/data/sprints/R9.0.plan.json`
- `references/data/sprints/R9.0.result.json`
- `reports/sprints/R9.0-baseline.md`
- `reports/sprints/R9.0-result.md`
- `reports/sprints/R9.0-diff-summary.md`
- `references/reference-team-roadmap.md`
- current guidance/docs that use Module 3 as a generic example
- `references/authored/` wording where Module 3 is only a sequencing shorthand
- generated reports, inventories, and RAG chunk data when regeneration is required after authored-reference wording changes

## Forbidden paths

- hand edits to `references/machine/`
- hand edits to `references/external/`
- deletion or refactor of the legacy source-data folder
- deletion or refactor of the legacy content-builder folder
- deletion or behavioral change of `deploy:legacy`
- student diagnostics
- adaptive routing
- student-facing AI
- automatic lesson sequencing, mastery decisions, or summative decisions

## Inputs

- `references/reference-team-roadmap.md`
- `AGENTS.md` and `CLAUDE.md` frozen legacy-target guardrails
- `BUILD-PARAGRAPH.md`
- current `references/authored/` files
- current `skills/` guidance
- `rg` scan for `Module 3`, `module-3`, and related legacy path labels

## Outputs

- Updated roadmap entry for R9.0.
- Reduced Module 3 references in non-functional guidance and authored course-design notes.
- Preserved functional legacy references where they point to actual legacy builders, deploy aliases, test harnesses, or guardrails.
- R9.0 sprint result logs and a report of remaining Module 3 references by category.

## Operationalized sprint procedure

1. Establish a baseline with a repository-wide `rg` scan and classify each Module 3 reference as functional legacy path, frozen-target guardrail, historical planning note, generated artifact, or safe wording cleanup.
2. Stop before editing any protected surface or functional legacy path. Do not change `references/machine/`, `references/external/`, legacy source-data/content-builder folders, or deploy behavior.
3. Apply only wording/reference cleanup in current guidance, authored course-design notes, and generic skill examples so new work points to book/source, blueprint, or later course-sequence language instead of Module 3.
4. Regenerate affected inventories/RAG chunks where authored-reference changes affect derived data, then run validators.
5. Record the result, remaining references, and acceptance test outcomes. If validation fails or a reference cannot be safely classified, leave it unchanged and report it for human review.

## Rollback plan

If the cleanup overreaches, revert the R9.0 commit. Functional legacy Module 3 paths are intentionally left untouched, so rollback should be limited to documentation, authored-reference wording, and regenerated derived data.

## Human review required

No new human gate is required. Report remaining Module 3 references after cleanup so the user can decide whether any historical or functional references should be handled later.
