# Sprint R4.3: Blueprint Flag Triage

## Goal

Convert raw `missing_units_flagged` entries from the target-exercise blueprint into a curated backlog with decision category, evidence, priority, and next action.

## Context

R4.2 produced a non-mutating exam-question extraction gap queue. R4.3 now handles the separate target-exercise backlog in `references/authored/course-target-exercises.json`.

This sprint is non-mutating. It must not mint units or edit protected machine references. It should finish the triage pass tightly by making every raw flag visible and dispositioned, so later work can decide what to mint, merge, defer, or reject through the proper CLI and review path.

## Outputs

- Curated blueprint flag backlog in JSON

- Sprint baseline/result/diff artifacts
- Roadmap update marking R4.3 complete and R5.1 next

## Operationalized sprint procedure

1. Establish baseline counts: number of target exercises, exercises with raw flags, total raw flags, unique raw flags, and current unit catalog size.
2. Build a deterministic checker that reads all raw flags and writes a triage record for every flag occurrence.
3. Classify each flag into the roadmap categories: `still_needed`, `existing_unit_match`, `duplicate`, `defer`, `reject`, or `needs_evidence`. Preserve the handoff category vocabulary as `handoff_status`.
4. Attach evidence for every decision: paragraph id, source reference, target-exercise prompt snippet, current required skills, best current unit matches, and the raw flag text.
5. Assign a priority and next action. Target-exercise-backed flags may become unit-creation or unit-scope-review candidates, but no unit mutation is allowed in R4.3.
6. Validate that every raw flag has exactly one triage record and that no protected reference file changed.
7. Write result artifacts and update the roadmap. Stop before any CLI mutation or authored-source cleanup.
