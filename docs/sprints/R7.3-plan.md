# Sprint R7.3: Retrieval Evaluation Set

## Goal

Create a fixed retrieval evaluation set and repeatable evaluation runner so RAG retrieval regressions are visible before any teacher-facing or broader internal use.

## Context

R5.3 allows internal retrieval development and evaluation only when source authority, edge status, diagnostic-only flags, pending-review flags, and generated-report warnings remain visible. R7.2 created the internal query prototype. R7.3 turns the required review questions into deterministic checks.

## Operationalized sprint procedure

1. Establish baseline: R7.2 query works, chunk index exists, and no fixed evaluation set/result exists yet.
2. Add natural-language text query support only as an internal evaluation convenience while preserving the existing unit, term, exam-code, and quality-issue modes.
3. Create the required ten retrieval evaluation cases, including expected entities/source types where stable and explicit warnings for diagnostic-only, pending-review, and generated-report behavior.
4. Implement a deterministic evaluation runner that executes each case through the R7.2 query layer, records found entities/source types/warnings, and writes JSON plus Markdown summaries.
5. Implement a validator that checks eval-set shape, result shape, authority handling, warning visibility, and that generated-report results are not treated as primary evidence.
6. Run the acceptance tests and stop if retrieval hides authority status, treats generated reports as primary evidence, or cannot expose diagnostic/pending-review warnings.
7. Write result artifacts and update the roadmap. R7.4 may review the behavior, but no RAG gate may be closed in R7.3.
